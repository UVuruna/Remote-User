## What This Project Is

Remote control of the Windows PC from an Android tablet/phone over LAN. The PC runs a Python server (screen capture + input injection); the tablet runs **no native app** — it opens a web page served by the PC. Full architecture and design decisions: [Remote User](README.md). Phase status: [Roadmap](ROADMAP.md).

## Tech Stack (decided — do not relitigate without owner approval)

**The authoritative record of the product direction and build plan is [ROADMAP.md](ROADMAP.md#direction) (Product Direction, decided 2026-07-22). Read it before proposing architecture changes.** Summary:

- **Server:** Python 3.13, FastAPI + uvicorn, `dxcam` (DXGI capture), `ctypes` → Win32 `SendInput` (injection), `qrcode` (pairing). Two entry points around one `server_core`: `gui_main.py` (PySide6 window + tray — what the packaged EXE runs) and `main.py` (headless CLI). `bootstrap.init_process()` (DPI → logging → user settings) MUST run before any screen-touching import.
- **Streaming:** **H.264** via ffmpeg, hardware-encoded with auto-detection (NVENC → QuickSync → AMF) and a **libx264 software fallback** so it runs on any PC; output is fragmented MP4 decoded by the client via MSE, at **native monitor resolution** by default (`h264_max_width`). JPEG-per-frame is the fallback when no ffmpeg/encoder exists. (Measured ~3.6 Mbps static native 4K vs ~37 Mbps JPEG at quarter resolution.) Bundled ffmpeg is **pinned to 7.1.1** — newer builds require NVENC API 13.1 (driver ≥ 610) and silently drop to software encoding on older drivers.
- **Product = two installable apps:** a Windows desktop app (server + GUI + tray, `.exe`) and an Android **hybrid** app (native Kotlin shell in `android/` + WebView reusing the web client, shipped as an APK; the shell adds only what a browser cannot — pairing, real-app link routing, file chooser, `Android.rescan()` / `Android.setTailscaleUrl()` bridge — ALL guidance lives once, in the page). The shell stores **two addresses** (LAN from pairing + Tailscale learned from every `config` via the bridge) and probes `/ping` on both at start, loading whichever answers — the app connects at home AND on mobile data with no user choice (a single stored URL was the live failure: LAN address on mobile data = minutes of `ERR_CONNECTION_TIMED_OUT`). **An Android browser NEVER shows the client** (owner rule 2026-07-22, shouted: no half-working browser sessions, no "Get the app" pill over a live page): the server routes Android browsers by User-Agent (the APK's WebView appends a `RemoteUserApp` marker) to `client/install.html` — a full-screen funnel whose FIRST, primary button is **Open the app** (`intent://pair?url=…` hands the tokened URL to the app; it pairs itself — nothing typed, nothing scanned; a camera-scanned QR always lands in the browser first, an Android platform limit for plain-http links, so every post-install scan must be exactly one tap), with the one-time Install (`/app.apk`) below it. After install the QR is not part of daily use — the user just opens the app; re-pairing happens IN the app. Desktop browsers still get the client (dev/testing). The desktop installer bundles the APK.
- **Connection:** a mesh VPN installed on both devices (Tailscale recommended), one-time, wizard-guided; works anywhere incl. mobile data via the mesh's free relay. Server already binds all interfaces + pairing detects the Tailscale address, so this is code-ready.
- **Login = the Tailscale login** (owner decision, option a): the mesh identity IS the account; one login per device, no custom account backend.
- **The app installs/drives all dependencies — the user never side-installs anything** (hard owner requirement). Wizard in the `.exe` AND the APK: ffmpeg bundled; Tailscale chain-installed on desktop / store-deep-linked-and-detected on Android; user only follows in-app prompts. Never tell the *end user* to download/copy/configure anything (telling the *owner* to install a dep for dev-time testing is fine, but say so explicitly).
- **Client UI:** vanilla HTML/CSS/JS in the WebView, Pointer Events, canvas rendering — no framework, no build step.
- **Hard constraints (owner):** no payment for any required part; third-party services/installs OK if best; not over-indexing on privacy/security. Don't fight mature remote-desktop tools on latency — the unique value is the **app-aware companion layer** (Windows UI Automation state reading, notifications, per-app controls).
- **Open decisions** (see ROADMAP): mesh provider (Tailscale vs ZeroTier), distribution (APK vs later Play Store), build order.

## Session Workflow (owner decree 2026-07-22)

**ALWAYS finish a session that touches server/client/android code by running the FULL desktop build** (`setup/build.py` — it bundles the client and the freshly built APK into the installer). The owner tests ONLY the real installed app, never the dev server from VSCode — shipping changes without a build means the owner installs stale software and the fixes "don't work". Build the APK first (`setup/build_apk.py`) when android/ or client/ changed. Do this without being asked; only GIT RELEASE still requires the owner's yes.

## Architecture Constraints

1. **One monitor per view.** The client always displays and controls exactly one monitor; a switch button changes which. Client coordinates are ALWAYS normalized 0–1 within the displayed monitor; the server maps them to that monitor's rect. Never send virtual-desktop-wide coordinates.
2. **The server process must declare DPI awareness** (`DPI_AWARENESS_CONTEXT_PER_MONITOR_AWARE_V2`) before any capture or injection — otherwise Windows silently rescales injected coordinates.
3. **No input before auth.** The WebSocket handler rejects every message until a valid token arrives. This is a hard security rule (Remote Mouse CVE class), not a nice-to-have.
4. **Unicode text goes through `KEYEVENTF_UNICODE`** (VK_PACKET), never through virtual-key mapping. Special keys (Enter, Backspace, arrows, Tab, Esc) go through VK codes. Two distinct protocol messages: `key_text` vs `key_special`.
5. **Client keyboard capture:** hidden focusable `<textarea>` (`opacity:0`, NOT `display:none` — a textarea so the phone IME shows ↵/new-row instead of a Send/Go key) with `autocapitalize/autocorrect/autocomplete/spellcheck` disabled; printable characters via input-event diffing, structural keys via `keydown`. Never trust `keydown.key` for printable characters (Android IME).
6. **The finger steers, buttons act** (owner decision 2026-07-22, replacing tap-to-click): the default one-finger gesture ONLY moves the PC cursor — it never clicks (tap-to-click was flaky on real devices and made every touch risky). Left clicks come from the explicit **Click** button (`click` message — down+up at the current cursor position; pressing it twice fast = double click). Toggle modes (right/drag/scroll) stay; two fingers on the canvas = local pinch zoom/pan, which never sends clicks. No timed gestures, ever.
7. **Ghost-pointer self-healing** (live bug 2026-07-22): Android WebView occasionally loses a `pointerup`/`pointercancel`; a stale entry in the pointer map made EVERY later tap register as a pinch until page refresh ("input randomly stops working"). On every `pointerdown` with `isPrimary`, the client wipes pointer/pinch/primary state — a new primary pointer is the browser's guarantee no other finger is really down. Do not remove this reset.
8. **Session lives only while the owner is looking** (owner security decision): the client closes the WebSocket when the page is hidden (tab backgrounded / screen locked) and reconnects **immediately** on return (`visibilitychange` → connect, not just the 2 s retry interval — the wait swallowed the first taps after every app switch). UAC-elevated windows still ignore injected input unless the server runs elevated (silent failure — UIPI).

## Protocol

- Client → server (JSON text): `auth {token}`, `pointer_down/pointer_up {x, y, button}`, `click {button}` (down+up at the CURRENT cursor position, no coordinates — the Click button), `pointer_move {x, y}`, `scroll {x, y, ticks}`, `viewport {x, y, w, h}` (JPEG mode only — H.264 always streams the full frame), `key_text {text}`, `key_special {key}`, `chord {chord}`, `monitor_switch {}`, `screenshot {}`. Coordinates are always 0–1 within the displayed monitor.
- Server → client (JSON text): `config {monitor_width, monitor_height, stream, codec?, tailscale_url, app_version}` — after auth and after EVERY stream (re)start (monitor switch, H.264 session reset); the client must fully reset its view and decode pipeline. `stream` is `"h264"` or `"jpeg"`; `codec` (H.264 only) is the exact MSE string (e.g. `avc1.4D4028`) parsed server-side from the live init segment's `avcC` box — never guessed; `tailscale_url` is the works-anywhere address with token (null until the PC is signed in to Tailscale) and feeds the client's guided wizard; `app_version` is the server's version — inside the APK the page compares it with the shell (`Android.appVersion()`) and offers the in-app update banner. `actions {sets}` (radial-wheel shortcut sets from `actions.json`, after auth), `toast {text}` (user-facing notice shown on the status pill), `cursor {x, y}` (PC pointer position, normalized, sent on change ~30 Hz — DXGI capture never contains the pointer; the client draws a virtual cursor in both modes).
- **Everything is guided IN the apps — never in chat/manuals** (hard owner principle, restated angrily 2026-07-22 and binding): any step a user must perform gets step-by-step on-screen guidance with plain language and automatic continuation. Desktop: three-state Tailscale card (install / sign in / connected). Phone first contact: the **QR always encodes the LAN address**, and the FIRST scan hard-requires the phone on the home Wi-Fi (a fresh phone has no tunnel to the PC — with mobile data the LAN address loads NOTHING, just a dead dark page; hit live 2026-07-22). The PC window states this requirement under the QR — the only place that can, since an unreachable page cannot guide anyone. Opening the QR in the phone browser lands on the **install funnel** (Open the app first/primary; Install below → paired automatically via `intent://` — the browser never shows anything else). Inside the app, the client page shows an "anywhere access" banner → in-page wizard: Google Play deep link for the Tailscale app → sign-in step → the page polls `GET /ping` (auth-free 204, reachability only) on the Tailscale address and hands over the permanent works-anywhere link the moment the phone joins. Never send the end user outside the guided flow.
- Server → client (binary), by announced mode: **H.264** — the raw fragmented-MP4 byte stream (init segment first; the client appends chunks into MSE in arrival order). **JPEG** — 16-byte header (4 × float32 LE — monitor-normalized x, y, w, h of the region the frame covers) + JPEG bytes.
- **H.264 streaming is per-client**: one shared capture feeds one ffmpeg process per connected client (fresh init segment + keyframe at connect — no mid-stream joining). H.264 bytes are never dropped individually; a client that falls a full queue behind gets its whole session reset (new `config` + stream). Capture/encode run only while ≥1 client is connected.
- **Controls are two configurable D-pad groups** (owner design): each group shows one category of up to 4 buttons, arranged as a D-pad cross in landscape and a column in portrait. A small dashed centre button opens a **tap-based** category wheel (tap to open, tap an item, ✕ cancels — no hold/drag). Top-left **Move** (pan, no click) and top-right **Hide** (hide all controls) are fixed. Everything — mouse modes, keyboard, monitor, snap, and chord shortcuts — is a category button defined in the owner-edited `actions.json` (see [ACTIONS.md](ACTIONS.md)), re-read every connection; the client renders from config and never hardcodes the layout. Custom-set editing UI and login belong to the future desktop GUI (owner decision).
- **Mouse modes are toggles, not holds** (owner correction): a single active `touchMode` — `move` (default: cursor only, never clicks) · `right` · `drag` · `scroll` · `pan` (the top-left Move) — decides what one finger does. Tapping a mode button toggles it (again → back to `move`); only one is ever active; two fingers always pinch-zoom. Left clicks are the explicit **Click** button (see Architecture Constraint 6). Do not reintroduce hold-to-activate, and do not reintroduce a separate `hover` mode — cursor-only IS the default.
- **Keyboard capture is invisible** (owner decision 2026-07-22, reversing the visible top bar — the bar only mirrored what the PC screen already shows): the `<textarea>` stays `opacity:0`; typed/dictated text is watched in the focused box on the streamed PC screen itself. The IME's ↵ key and IME-committed `"\n"` both become **Shift+Enter** (new row — messengers keep typing instead of sending); the real Enter (activate/send) is the D-pad **Enter** button (`key_special enter`) in `actions.json`.
- **Phone → PC image** (`upload` action, owner request): a phone file-picker (gallery/camera) POSTs the image to `/upload` (token-gated); the server decodes it, puts it in the **PC clipboard** and **injects Ctrl+V itself** — picking the image is the whole gesture (owner 2026-07-22: the user already clicked the target box; a manual paste step was dead weight). Decoding is Pillow-first with `pillow-heif` registered (phone cameras default to **HEIC**, which OpenCV cannot read — found live) and applies EXIF orientation; cv2 is the fallback. This is the opposite of `snap` (which captures the PC screen); the owner wants phone→PC, so `snap` is not in the default layout.
- **Updates flow downhill: GitHub → PC → phone** (owner request 2026-07-22). Only the DESKTOP app checks the internet: once per start it asks the public repo's latest GitHub release (`updates.py`, `update_repo` in config) and shows an in-window "Update to vX" button — download installer, launch, quit. The PHONE never checks the internet: `config.app_version` tells the page what the PC runs, the `Android.appVersion()` bridge tells it what the shell is, and a newer PC shows the "Update the app" banner → `Android.update()` downloads `/app.apk` from the SAME PC via the system browser. Publishing a GIT RELEASE (tag + `RemoteUser_Setup.exe` asset) is what lights the desktop path up.
- **Rendering is two layers** (owner request — smooth pan/zoom): a full-monitor **base** frame kept in memory (drawn always, so motion never flashes blank) plus the sharp **region** crop drawn on top when zoomed. Full native resolution is never streamed live (bandwidth); the base is the last full frame, the region refreshes while zoomed.
- Region streaming: when zoomed, the client requests its visible region (`viewport`); the server crops the 4K frame to it before downscaling — zoom stays sharp at constant bandwidth. Full frame = region (0, 0, 1, 1).

## Testing Notes

- Gesture disambiguation (tap vs drag vs scroll) can only be tuned on a **real touch device** — do not assume defaults are right.
- Test Unicode injection against real targets (VSCode, browser inputs) — VK_PACKET has known quirks in some apps (e.g. Windows Terminal surrogate-pair bugs).
- After long-running capture sessions, verify frame latency fresh — restart before measuring (see root CLAUDE.md profiling section).
